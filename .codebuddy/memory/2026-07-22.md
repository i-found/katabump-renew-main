## Turnstile 验证码调试全过程（重要经验）

- 部署后持续卡在 Cloudflare Turnstile，经过多轮调试：
  1. playwright npm 版本与 Docker 镜像版本不匹配 → 锁定两边均为 v1.61.1
  2. 点击坐标偏移到容器中心（容器被 EXPAND_JS 撑大到 1292px）→ 改点左侧 checkbox 位置仍失败
  3. 发现 challenges.cloudflare.com frame 内部为空 → checkbox 在 Shadow DOM 中
  4. 普通 querySelector/iframe API 找不到 Shadow DOM 内的 checkbox
  5. **关键突破**：参考用户原项目 `i-found/katabump`（renew.js），其能跑的核心技术：
     - `INJECTED_SCRIPT` Hook `Element.prototype.attachShadow`，拦截 Turnstile 创建 Shadow DOM 瞬间，捕获 checkbox 坐标比存入 `window.__turnstile_data`
     - `attemptTurnstileCdp(page)` 遍历 frames 读 `__turnstile_data`，用 **CDP `Input.dispatchMouseEvent`** 发原生鼠标事件（比 Playwright page.mouse.click 更难被检测）
  6. CDP 点击发出后坐标正确，但仍失败 → **根因：headless 模式被 Cloudflare 检测**
  7. **最终修复**：Dockerfile 安装 xvfb，`xvfb-run -a -s "-screen 0 1920x1080x24" node index.js` + index.js `headless: false`，让 Chrome 在有头虚拟显示器下运行（对齐原 Python 版 GitHub Actions 的 xvfb-run 做法）

## 经验教训
- Cloudflare Turnstile 在 headless 容器里极难通过，必须用 xvfb + 有头模式 + CDP 原生点击 + attachShadow Hook
- 用户的 `i-found/katabump` 是已验证可用的 Node.js 参考实现，遇到瓶颈应先参考它

## 修复容器卡在 "Starting Container"（xvfb-run 不可靠）
- 现象：改用 `xvfb-run -a ... node index.js` 后容器一直卡在 Starting Container，疑似 Railway 环境下 xvfb-run 启动/挂起导致容器无法进入健康状态、反复重启
- 诊断：index.js 本身健壮（startHealthServer 在浏览器启动前就监听 $PORT，所有错误均被捕获，进程不会退出），卡住纯粹是 Dockerfile CMD 的 `xvfb-run` 包装器问题
- 修复方案：
  - 新建 `start.sh`：显式 `Xvfb :99 -screen 0 1920x1080x24 -nolisten tcp &`，等待 `/tmp/.X11-unix/X99` 套接字就绪（最多 10s），`export DISPLAY=:99`，最后 `exec node index.js`；并先 `mkdir -p /tmp/.X11-unix` + `rm -f /tmp/.X99-lock` 防残留锁
  - Dockerfile: `RUN chmod +x start.sh` + `CMD ["./start.sh"]` 替代 `xvfb-run`
  - index.js `launchBrowser()` 增加回退：检测 `process.env.DISPLAY`，有则 `headless:false`（有头），无则 `headless:true`（headless 回退），避免 Xvfb 异常时崩溃循环
